# Skill: Unit Tests — FIRST Principles

## Purpose

This skill defines the **FIRST** principles that every unit test generated by the Unit Test Generator agent must satisfy. Apply this checklist to each test before finalising it.

---

## FIRST Principles

### F — Fast
Tests must run in milliseconds.

**Rules:**
- No file I/O, database calls, or network requests
- No `Thread.Sleep` or `Task.Delay`
- Use in-memory fakes or stubs for any external dependency
- A full test suite should complete in under 5 seconds

### I — Independent
Tests must not depend on each other or on shared mutable state.

**Rules:**
- Each test creates its own fresh instance of the class under test
- No static mutable fields shared between tests
- Test execution order must not matter — any test must be runnable in isolation
- Use `[Fact]` (xUnit) or equivalent; avoid shared constructor side-effects that mutate state

### R — Repeatable
Tests must produce the same result on every run, in any environment.

**Rules:**
- No `DateTime.Now` / `DateTime.UtcNow` — inject or stub time if needed
- No `Random` or `Guid.NewGuid()` in assertions
- No dependency on environment variables, file system paths, or OS locale
- No reliance on external services or network

### S — Self-Validating
Tests must produce an unambiguous pass or fail without human inspection.

**Rules:**
- Every test must contain at least one assertion (`Assert.Equal`, `Assert.True`, etc.)
- No `Console.WriteLine` as a substitute for assertions
- Failure messages must be descriptive enough to diagnose the problem without reading the code
- Do not catch exceptions silently — let unexpected exceptions fail the test

### T — Timely
Tests must be written for the code being changed now — not speculatively.

**Rules:**
- Only generate tests for methods or code paths identified in `fix-summary.md`
- Do not write tests for unchanged code
- Do not write tests for hypothetical future features
- Tests should reflect the actual fix applied, covering both the buggy scenario (now expected to pass) and the correct behaviour

---

## FIRST Compliance Checklist

For every test generated, verify each principle before including it in the output:

| Principle | Check |
|-----------|-------|
| **F**ast | No I/O, no sleep, no network — runs in ms |
| **I**ndependent | Fresh state per test, no shared mutation |
| **R**epeatable | No time/random/env dependencies |
| **S**elf-validating | Has assertions, failure is unambiguous |
| **T**imely | Only covers code changed per fix-summary |

Mark each ✓ (satisfied) or ✗ (violation found — fix before including the test).

---

## Required Output Sections in `test-report.md`

The Unit Test Generator **must** produce a report with exactly these sections:

1. `## Summary` — total tests generated, passed, failed, skipped
2. `## Tests Generated` — table: test name | file | method under test | FIRST notes
3. `## Test Results` — raw or summarised output from the test runner
4. `## FIRST Compliance` — per-test checklist (each of F/I/R/S/T marked ✓ or ✗)
5. `## References` — test files created, fix-summary used, this skill
